## 需求

1. 在修改资源时，实现前端页面的实时刷新。在手机端进行修改时，可以同步到电脑端的前端。
2. 实现 `data/state.json` 的变更历史记录（只记录 state 的变化，不包含爬虫）。
3. 撤销：支持误操作撤销（Undo），并且撤销本身也可追踪。
4. 接入爬虫修改。

## 已完成功能

1. 保存 DP 时：只在 DP 真变化时写入 `data/state.json`，并追加一条 `data/state_history.jsonl` 记录。
2. 前端自动更新：浏览器通过 SSE 连接 `/api/state-events`，当后端发现 `data/state.json` 有变化时通知前端重新读取。
3. 撤销（Undo）：新增 `/api/undo`，撤销最近一条未被撤销过的历史记录，并把撤销本身追加为一条 `undo` history。
4. 接入爬虫修改：服务端在爬虫运行结束后（以及启动时补偿一次），会读取 `data/crawler_state.json` 的 `pending_dp_*`，把扣除值正式写入 `data/state.json`，并记录一条 `crawler_dp_apply` history（带 `pending_dp_id` 防重复）。

## 文件拆分约定

- `data/state.json`：只保存“当前状态数据”（例如 dp、gp、未来的 inventory/quests 等）。不保存任何历史记录。
- `data/crawler_state.json`：只保存爬虫运行状态（独立文件）。爬虫排除在 state 历史之外。
- `data/state_history.jsonl`：保存 `data/state.json` 的变更历史（JSON Lines：一行一条 JSON）。

## history 记录格式（schema v1）

每条历史记录是一行 JSON，包含两个核心部分：

1) `data`：事件上下文（发生了什么、原因是什么）。
2) `changes`：这次事件对 `data/state.json` 造成的实际变化（from/to）。

字段建议：

- `v`：schema 版本号，初始为 1。
- `ts`：Unix 时间戳。
- `text`：可读时间字符串（便于人眼查看）。
- `type`：事件类型（可扩展，例如 `dp_set`、`item_gain`、`item_spend`、`quest_update`、`undo` 等）。
- `actor`：触发来源（例如 `server`、`manual`、未来的脚本名）。
- `note`：一句话说明（可选）。
- `data`：上下文信息（按 A 方案：尽量少、尽量不重复 state 的结果）。
- `changes`：数组，元素格式为 `{ "path": "...", "from": <旧值>, "to": <新值> }`。

示例（物品获得，A 方案：不在 data 里重复数量）：

```json
{"v":1,"ts":1700000200,"text":"2026-03-11 10:00:00","type":"item_gain","actor":"server","note":"打怪掉落","data":{"item_id":"potion_small","source":"slime"},"changes":[{"path":"inventory.potion_small","from":5,"to":7}]}
```

## path 规则（点号路径）

- `changes.path` 使用点号字符串表示层级：`inventory.potion_small` 表示 state.json 的 `inventory` 下的 `potion_small` 字段。
- 约定：用作 key 的字符串不允许包含 `.`（避免路径歧义）。

## 只记录“有变化”的事件

- 当且仅当 `data/state.json` 的值发生变化时，才写入一条 history。
- 重复保存同一个值，不写 history。

## 撤销（Undo）设计

- 撤销的本质是一次新的事件（不改写旧历史）。
- Undo 逻辑：找到最近一条“可撤销”的记录，把其中每个 change 反向应用（把值写回 `from`）。
- 撤销成功后，追加一条新的 history，`type` 为 `undo`，并在 `data` 中写明撤销了哪一条（例如 `undo_of_ts`）。

示例（撤销某条记录）：

```json
{"v":1,"ts":1700000300,"text":"2026-03-11 10:05:00","type":"undo","actor":"server","data":{"undo_of_ts":1700000200},"changes":[{"path":"inventory.potion_small","from":7,"to":5}]}
```

## history 变多的处理方式

- 保留 undo 记录（保证可追溯）。
- 展示时可做折叠/过滤（例如把“原操作 + undo”显示成“已撤销”）。
- 可加简单的归档/轮转策略（例如只保留最近 N 条，旧的移动到 archive）。

## 原子写入（防止文件写坏）

- 写 JSON 文件时优先采用“先写临时文件 -> 再替换原文件”的方式。
- 目的：避免程序中断导致出现“半截 JSON”。
